DATA FLOW PATTERNS
================================================================================

Assessment Output:
  assess-review-issues.sh outputs assessment to stdout (pipe-friendly)
  assess-and-resolve.sh captures stdout, decides to loop (exit 2) or merge (exit 0)
  workflow-runner.sh captures exit codes and stdout to pass review content to fix mode
  stderr is used for all user-facing output (print_info, print_warning, etc.)

Review Sensitivity:
  Detected in create-pr.sh early checks (informational)
  Injected into review prompt by local-review.sh via detect_sensitivity_areas()
  Patterns configured in .rite/blockers.conf (BLOCKER_* variables)

Exit Codes (semantic):
  0   = Success, ready to proceed
  1   = Failure, manual intervention needed
  2   = Loop needed (e.g., fixes found, re-review required)
  3   = Review stale, route back to Phase 2
  5   = Provider usage cap reached (blocks batch processing)
  10  = Stale branch restart, skip resume state
  124 = Timeout (from gtimeout/timeout)
  127 = Command not found

Config Loading Order:
  1. config.sh sets defaults
  2. ~/.config/rite/config sourced (global, optional)
  3. ~/.riterc sourced (convenience, optional)
  4. $RITE_PROJECT_ROOT/.rite/config sourced (project-specific, optional)
  5. $RITE_PROJECT_ROOT/.rite/blockers.conf sourced (blocker patterns, optional)
  6. Environment variables override everything

Logging:
  Structured [phase] and [diag] lines logged to RITE_LOG_FILE
  Used by health-report for aggregation and diagnostics
  Auto-pruned: keeps last 20 logs, deletes older

WORKFLOW PHASES
================================================================================

Phase 1: Development
  Script: lib/core/claude-workflow.sh
  Provider: RITE_DEV_PROVIDER (default: claude)
  Input: ISSUE_NUMBER, WORK_DESCRIPTION
  Output: Code changes in worktree
  Exit codes: 0=success, 1=fail, 5=quota

Phase 2: Push/PR
  Script: lib/core/create-pr.sh
  Input: Worktree with commits
  Output: PR created/updated on GitHub, early sensitivity detection
  Exit codes: 0=success, 1=fail, 2=already exists

Phase 2b: Review Generation
  Script: lib/core/local-review.sh
  Provider: RITE_REVIEW_PROVIDER (default: claude)
  Input: PR diff
  Output: Review comment posted to PR
  Sensitivity hints injected from blocker detection

Phase 3: Assessment Loop (up to 3 retries)
  Script: lib/core/assess-review-issues.sh + assess-and-resolve.sh
  Provider: RITE_REVIEW_PROVIDER
  Input: Review findings
  Output: Three states: ACTIONABLE_NOW, ACTIONABLE_LATER, DISMISSED
  Decisions: Loop (fix), Create follow-up, or Merge

Phase 4: Merge
  Script: lib/core/merge-pr.sh
  Input: PR with passing review
  Output: PR merged, branch deleted, worktree cleaned
  Post-merge: Run tests, attribute failures

Phase 5: Completion
  Notifications, cleanup, health tracking

================================================================================
SAFETY SYSTEM (Two-tier)
================================================================================

Tier 1: Review Sensitivity Hints (non-blocking)
  Detected: Infrastructure, migrations, auth, docs, expensive services
  Purpose: Focus reviewer attention on sensitive areas
  Effect: Injected into review prompt, not a hard gate

Tier 2: Hard Merge Gates (blocking)
  CRITICAL review findings → requires fix or approval
  Test/build failures → non-zero exit blocks merge
  Session limits exceeded → stops workflow
  AWS credentials expired → deployment credentials invalid
  Blocker approvals → remembered per-issue via has_approved_blocker()
  Supervised mode: interactive read -p prompt
  Unsupervised mode: stops workflow (unless --bypass-blockers)

Stale Branch Handling:
  Below threshold: auto merge origin/main into feature branch
  At/above threshold (auto): close PR with summary, cleanup, restart fresh
  At/above threshold (supervised): prompt with 4 options

================================================================================
PROVIDER AGNOSTICISM (Critical Rules)
================================================================================

All prompts (review, assessment, planning) are provider-agnostic plain Markdown
Provider-specific behavior isolated in lib/providers/<name>.sh behind 17-function interface

Rules:
  - No prompt may contain provider-specific instructions (Claude /exit, tool_use, --disallowedTools)
  - Provider-specific instructions go in preamble functions
  - Model names are metadata only, never instructional text
  - Error patterns, tool restrictions, streaming format all provider-specific
  - Per-phase provider selection: RITE_DEV_PROVIDER, RITE_REVIEW_PROVIDER, RITE_UTILITY_PROVIDER

Prompt Design:
  1. Sharkrite identity: "You are running inside a Sharkrite workflow session"
  2. Git/GH prohibition: Enforced via BOTH prompt AND --disallowedTools
  3. TodoWrite restriction: Blocked, causes performative "phases" instead of work
  4. Explicit exit: Supervised mode needs /exit, auto mode uses --print
  5. No open-ended questions: Directives, not "Ready to start?"

SHELL CONVENTIONS (CRITICAL)
================================================================================

grep -c pattern behavior:
  BAD:  COUNT=$(echo "$text" | grep -c "pattern" || echo "0")  # outputs "0\n0"
  GOOD: COUNT=$(echo "$text" | grep -c "pattern" || true)      # grep already outputs count

grep -o vs grep -c:
  grep -c always outputs count (even "0"), returns exit 1 on 0
  grep -o outputs nothing on no match, so || echo "0" is needed there

Structured header matching (CRITICAL):
  BAD:  grep "ACTIONABLE_NOW"      # matches anywhere, including reasoning text
  GOOD: grep "^### .* - ACTIONABLE_NOW"  # matches only classification headers

Review severity parsing:
  BAD:  grep -c "CRITICAL:"         # matches "Findings: [CRITICAL: 0 | ...]"
  GOOD: grep "CRITICAL: [0-9]+"    # matches only the Findings line

Unbound variables with set -u (CRITICAL):
  BAD:  if [ -z "$WORKTREE_PATH" ];
  GOOD: if [ -z "${WORKTREE_PATH:-}" ];

Exported env vars vs function definitions:
  Exported vars survive subprocesses, function definitions don't
  Don't use env var as skip guard for source if child processes need the functions

PIPESTATUS survival:
  BAD:  OUTPUT=$(cmd1 | cmd2); EXIT_CODE=${PIPESTATUS[0]}  # PIPESTATUS from outer shell
  GOOD: Use temp file inside pipeline to capture exit code

local keyword usage:
  local only works inside functions
  Main script body (not in function): use _prefix assignment instead
  Crashes in main body: "local: can only be used in a function"

git push (CRITICAL):
  BAD:  git push              # relies on upstream tracking, may push to origin/main
  GOOD: git push origin "$BRANCH_NAME"  # explicit remote and branch

.gitignore and symlinks:
  Use `.rite` (no trailing slash) — `.rite/` only matches directories,
  but in worktrees `.rite` is a symlink (git mode 120000)

TOKEN OPTIMIZATION: rtk (Trial)
================================================================================

Status: Trial (installed 2026-03-24)
Tool: rtk - CLI proxy that compresses terminal output before Claude Code sees it

What rtk affects:
  - ONLY Claude Code Bash tool calls
  - Sharkrite's own scripts call git/gh/jq directly — rtk never touches them
  - Phase 1 (development) is where savings happen: git status, git diff, test runs, grep, cat
  - Phases 2-5 unaffected: all Sharkrite gh calls use --json which rtk passes unfiltered
  - stdin piping (fix-review mode) unaffected: hook only rewrites command strings

Configuration:
  ~/.config/rtk/config.toml           Global config
  .rtk/filters.toml                   Project-local filter overrides
  ~/.claude/hooks/rtk-rewrite.sh      PreToolUse hook

Excluded commands: cat, head, tail (strips code comments, causing style mismatches)

Diagnosing rtk issues:
  RTK_TOML_DEBUG=1 rtk git status     Shows which filter matched
  rtk stats                            Overall savings
  rtk stats --detail                   Per-command breakdown
  rtk init --global --uninstall       Temporarily disable
  rtk init --global --hook-only       Re-enable

Weekly health report:
  launchd job (com.sharkrite.health-report) runs Monday 9:07 AM
  Generates .rite/reports/rite-health-YYYYMMDD.md
  Uses absolute thresholds (not before/after comparison):
    - Fix iterations avg > 2.0 → WARNING
    - Any phase failing > 30% → WARNING
    - Phase 1 duration avg > 20 min → WATCH
    - rtk savings < 30% → WATCH

If rtk causes token waste: rtk init --global --uninstall && brew uninstall rtk
